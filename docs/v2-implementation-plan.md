# SuperClaw v2 Implementation Plan

## Overview

SuperClaw v2 将当前 5-skill 流水线升级为一个基于文件系统的任务管理平台，增加持久化 agent、分级治理、记忆整理和更强的对齐能力。核心理念不变：人管方向，OpenClaw 管对齐和验收，Claude Code 管执行。

v2 的八个组件：

1. **File-Driven Board** — 文件系统看板，取代隐式状态流转
2. **Phase Integration** — 五阶段 skill 与 board 的精确联动
3. **Delivery Tier T0-T3** — 风险分级治理
4. **AutoDream** — 记忆整理持久 agent
5. **Persistent Agent Framework** — 通用持久 agent 框架
6. **Align Skill Upgrade** — 借鉴 gstack office-hours 升级对齐能力
7. **Constraint Enforcement** — 约束自动化 lint
8. **Board Web UI** — 基于 Go + Next.js 的看板 Web 界面

---

## Component 1: File-Driven Board

### Current State

当前 SuperClaw 没有持久化的任务状态。任务状态隐含在 skill 流转中：align 产出 spec.md，plan 产出 plan.md，execute/verify/deliver 在对话 session 内完成。没有跨 session 的任务追踪，无法知道"哪些事在进行中"。

### Target State

一个纯文件系统的看板，目录 = 列，Markdown 文件 = 卡片。任何工具都能读写（OpenClaw、Claude Code、shell 脚本、人手动 `mv`）。

### File Structure

```
.superclaw/
├── board/
│   ├── inbox/            # 新任务，等待对齐
│   ├── aligning/         # align 阶段进行中
│   ├── planned/          # spec + plan 已完成，等待执行
│   ├── executing/        # Claude Code 执行中
│   ├── reviewing/        # verify 阶段
│   ├── done/             # 完成归档
│   └── blocked/          # 阻塞，需要人工介入
├── config/
│   ├── board.yaml        # board 全局配置（ID 计数器等）
│   └── tiers/            # T0-T3 checklist 定义
│       ├── t0.yaml
│       ├── t1.yaml
│       ├── t2.yaml
│       └── t3.yaml
├── agents/               # 持久 agent 任务文件
│   ├── autodream.md
│   ├── doc-sync.md
│   └── ci-watch.md
├── context/
│   └── project-context.md  # Phase 0 扫描产出
└── state/
    ├── last_event.json     # 最近事件（现有）
    └── tool_log.jsonl      # 工具日志（现有）
```

### Task File Spec

每个任务文件命名：`{NNN}-{slug}.md`，例如 `001-add-dark-mode.md`。

完整格式：

```markdown
---
id: "001"
slug: add-dark-mode
created: 2026-04-05T10:30:00+08:00
updated: 2026-04-05T14:20:00+08:00
assignee: openclaw          # openclaw | cc | human | agent:{name}
priority: high              # critical | high | medium | low
type: feature               # feature | bugfix | refactor | chore | spike
tier: T2                    # T0 | T1 | T2 | T3
phase: aligning             # inbox | aligning | planned | executing | reviewing | done | blocked
blocked_reason: ""          # 非空则在 blocked 列
parent: ""                  # 父任务 ID（子任务拆分场景）
spec_path: ""               # spec.md 路径（align 完成后填入）
plan_path: ""               # plan.md 路径（plan 完成后填入）
---

# Add Dark Mode

## Description

用户要求给管理后台增加深色模式支持。

## Acceptance Criteria

- [ ] 所有页面支持 dark/light 切换
- [ ] 用户偏好持久化到 localStorage
- [ ] 无 FOUC (Flash of Unstyled Content)

## Verify

```bash
# 命令 + 预期输出
npm test -- --grep "dark-mode"
# Expected: All tests pass, exit code 0
```

## History

| Time | Phase | Actor | Note |
|------|-------|-------|------|
| 2026-04-05T10:30 | inbox | human | 创建任务 |
| 2026-04-05T10:35 | aligning | openclaw | 开始对齐 |
```

### Task ID Generation

- `.superclaw/config/board.yaml` 中维护 `next_id: 2` 计数器
- 每次创建任务：读计数器 → 用当前值 → 计数器 +1 → 写回
- ID 格式：3 位零填充 (`001`, `002`, ... `999`)
- 超过 999 后自动扩展到 4 位

```yaml
# .superclaw/config/board.yaml
next_id: 2
default_tier: T2
default_priority: medium
```

### How Skills Trigger File Moves

移动任务 = `mv .superclaw/board/{from}/{NNN}-{slug}.md .superclaw/board/{to}/` + 更新文件内 `phase` 字段 + 追加 History 行。

封装为 shell 函数（可在 skill 和 hook 中调用）：

```bash
# .superclaw/bin/board-move.sh
# Usage: board-move.sh 001-add-dark-mode.md executing reviewing "verify 开始"
TASK_FILE="$1"
FROM="$2"
TO="$3"
NOTE="$4"
TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)

mv ".superclaw/board/$FROM/$TASK_FILE" ".superclaw/board/$TO/$TASK_FILE"
# 更新 phase frontmatter + 追加 history（用 sed 或脚本）
```

### Implementation Steps

1. 创建 `.superclaw/` 目录结构 + board.yaml 初始配置
2. 编写 `board-move.sh`、`board-create.sh`、`board-list.sh` 三个核心 CLI 工具
3. 编写 `board-status.sh` — 列出所有列的任务摘要（供 OpenClaw 调用）
4. 为每个 skill 添加 board 操作（Component 2 详述）
5. 编写测试：创建/移动/列表/边界情况

### Estimated Effort

L — 约 3-4 个执行 session。主要工作在 CLI 工具 + 测试。

## Component 2: Phase Integration

### Current State

五个 skill（align/plan/execute/verify/deliver）通过 `invoke` 链式调用，状态隐含在对话上下文中。spec.md 和 plan.md 保存位置不固定。

### Target State

每个 phase 对 board 有明确的读/写/移动操作。任务文件是唯一的状态源。

### Phase 0: Project Onboarding（新增）

**触发：** 首次在项目中使用 SuperClaw 时，或用户明确要求重新扫描。

**操作：**
1. 读取：`README.md`、`CLAUDE.md`、`package.json`/`Cargo.toml`/etc.、`git log --oneline -20`、目录结构
2. 写入：`.superclaw/context/project-context.md`
3. 不涉及 board 操作

**project-context.md 格式：**

```markdown
# Project Context

> Auto-generated by SuperClaw Phase 0
> Last updated: YYYY-MM-DD

## Project
- Name: xxx
- Language: TypeScript
- Framework: Next.js
- Package Manager: pnpm

## Structure
- Source: src/
- Tests: tests/
- Config: .config/

## Key Commands
- Build: pnpm build
- Test: pnpm test
- Lint: pnpm lint

## Recent Activity
- [最近 10 条 commit 的单行摘要]

## Architecture Notes
- [从 README/CLAUDE.md 提取的关键架构信息]
```

### Phase 1: Align

**Board 操作：**

| 步骤 | 操作 | 命令 |
|------|------|------|
| 开始 | 从 inbox 移到 aligning | `board-move.sh {task} inbox aligning "开始对齐"` |
| 写 spec | 保存 spec.md 到项目目录 | `write spec.md` + 更新任务 `spec_path` 字段 |
| 用户 approve | 从 aligning 移到 planned | `board-move.sh {task} aligning planned "spec approved"` |

**读取：** 任务文件（description、acceptance criteria）、project-context.md
**写入：** spec.md、更新任务文件（spec_path、history）

### Phase 2: Plan

**Board 操作：**

| 步骤 | 操作 | 命令 |
|------|------|------|
| 开始 | 读 planned 列中的任务 | 读任务文件 + spec_path |
| CC 出 plan | 保存 plan.md | `write plan.md` + 更新任务 `plan_path` |
| plan 确认 | 保持在 planned（不移动） | 更新 history |

**读取：** 任务文件、spec.md、project-context.md
**写入：** plan.md、更新任务文件（plan_path、history）

### Phase 3: Execute

**Board 操作：**

| 步骤 | 操作 | 命令 |
|------|------|------|
| 开始 | 从 planned 移到 executing | `board-move.sh {task} planned executing "开始执行"` |
| CC 执行中 | 定期更新 progress | 追加 history + 更新 `updated` 时间戳 |
| CC 完成 | 从 executing 移到 reviewing | `board-move.sh {task} executing reviewing "执行完成"` |
| CC 阻塞 | 从 executing 移到 blocked | `board-move.sh {task} executing blocked "原因: ..."` |

**读取：** 任务文件、plan.md、project-context.md
**写入：** 更新任务文件（history、blocked_reason）

**错误处理 — CC 崩溃/中断：**
1. Stop hook 写 `last_event.json`，包含 `status: "interrupted"` 字段
2. OpenClaw 下次检查时发现任务仍在 `executing/` 但 session 已结束
3. 读取 `tool_log.jsonl` 判断进度
4. 选择：恢复执行（`cc-delegate session continue`）、移到 blocked、移回 planned 重新规划

### Phase 4: Verify

**Board 操作：**

| 步骤 | 操作 | 命令 |
|------|------|------|
| 开始 | 读 reviewing 列任务 | 读任务文件 + spec + verify 命令 |
| 不通过 | 移回 executing | `board-move.sh {task} reviewing executing "验收不通过: ..."` |
| 通过 | 保持在 reviewing（等 deliver） | 更新 history |

**读取：** 任务文件、spec.md（acceptance criteria）、代码产出
**写入：** 验证报告、更新任务文件 history

### Phase 5: Deliver

**Board 操作：**

| 步骤 | 操作 | 命令 |
|------|------|------|
| 呈现 | 读 reviewing 任务 + 验证报告 | 生成交付报告 |
| 用户 approve | 从 reviewing 移到 done | `board-move.sh {task} reviewing done "用户 approved"` |
| 用户要调整 | 移回 executing 或 planned | `board-move.sh {task} reviewing {target} "用户要调整: ..."` |
| 用户丢弃 | 从 reviewing 移到 done | `board-move.sh {task} reviewing done "用户丢弃"` |

### Implementation Steps

1. 为 Phase 0 创建新 skill：`skills/onboard/SKILL.md`
2. 修改 align skill：增加 board 读写操作
3. 修改 plan skill：增加 board 读写操作
4. 修改 execute skill：增加 board 移动 + 崩溃恢复逻辑
5. 修改 verify skill：增加 board 移动 + verify 命令执行
6. 修改 deliver skill：增加 board 移动 + 归档
7. 修改 cc-delegate notify hook：在 Stop 事件时更新 board 状态
8. 编写集成测试：完整 inbox → done 流转

### Estimated Effort

L — 约 4-5 个执行 session。每个 skill 都需要修改，需要仔细测试状态流转。

## Component 3: Delivery Tier T0-T3

### Current State

所有任务走相同的验证流程，没有风险分级。一个修 typo 的任务和一个改支付逻辑的任务经过相同的验收。

### Target State

四级交付治理，风险越高，检查越严格：

| Tier | 适用场景 | 验证强度 |
|------|---------|---------|
| T0 | 生产核心路径（支付、认证、数据迁移）| 完整：TDD + E2E + 安全扫描 + 手动验收 + code review |
| T1 | 合作伙伴/预览功能 | 安全 + 可观测性 + 自动化测试 |
| T2 | 内部工具 | 基本：单元测试 + lint |
| T3 | 开发脚本/文档/配置 | 最小：语法检查 + 自审 |

### Tier Declaration

在任务文件 frontmatter 中声明：

```yaml
tier: T2    # T0 | T1 | T2 | T3
```

默认值由 `.superclaw/config/board.yaml` 的 `default_tier` 控制。用户可在 align 阶段手动调整。

### Tier Checklist 定义

每个 tier 有独立的 checklist 文件：

```yaml
# .superclaw/config/tiers/t0.yaml
name: "T0 — Production Core"
description: "生产核心路径，完整检查"
checklist:
  - id: tdd
    name: "TDD 测试全通过"
    verify: "npm test"
    required: true
  - id: e2e
    name: "E2E 端到端测试"
    verify: "npm run test:e2e"
    required: true
  - id: security
    name: "安全扫描"
    verify: "npm audit --production"
    required: true
  - id: lint
    name: "代码 lint"
    verify: "npm run lint"
    required: true
  - id: type_check
    name: "类型检查"
    verify: "npm run typecheck"
    required: true
  - id: manual_review
    name: "人工 code review"
    verify: null  # 人工操作，不可自动化
    required: true
  - id: observability
    name: "可观测性检查（日志、metrics、告警）"
    verify: null
    required: true
```

```yaml
# .superclaw/config/tiers/t3.yaml
name: "T3 — Dev Scripts"
description: "开发脚本、文档、配置，最小检查"
checklist:
  - id: syntax
    name: "语法/格式检查"
    verify: "npm run lint -- --fix"
    required: true
  - id: self_review
    name: "AI 自审"
    verify: null
    required: true
```

### Verify Phase 如何加载 Checklist

1. verify skill 读取任务文件 → 获取 `tier` 字段
2. 读取 `.superclaw/config/tiers/{tier}.yaml`
3. 逐项执行 `verify` 命令（有命令的项）
4. 人工项标记为 "待人工确认"
5. 所有 required 项通过 → 验收通过
6. 任一 required 项失败 → 验收不通过，列出失败项

### Implementation Steps

1. 设计 tier YAML schema
2. 创建 4 个默认 tier 配置文件
3. 修改 verify skill：按 tier 加载 checklist
4. 修改 align skill：在对齐阶段提示用户选择/确认 tier
5. 编写 `tier-check.sh`：给定 tier 配置运行所有可自动化项
6. 编写测试

### Estimated Effort

M — 约 2 个执行 session。主要是 YAML 定义 + verify skill 修改。

## Component 4: AutoDream — Memory Consolidation

### Current State

OpenClaw 有 memory 目录（`~/.openclaw/memory/`），包含 `MEMORY.md` 和若干主题文件。但这些文件只在对话中被手动更新，没有自动整理机制。随着使用增多，内容重复、过时、碎片化。

### Target State

AutoDream 是第一个持久 agent，定期整理 OpenClaw 的记忆文件。它是一个自主运行的后台任务，不需要用户触发。

### Trigger Detection

**触发条件：** 两个条件同时满足：
1. 距上次 consolidation ≥ 24 小时
2. 自上次 consolidation 以来 ≥ 5 个 session

**追踪文件：** `.superclaw/agents/autodream-state.json`

```json
{
  "last_consolidation": "2026-04-04T03:00:00Z",
  "sessions_since_last": 7,
  "last_session_id": "abc123"
}
```

**Session 计数更新：** OpenClaw 的 heartbeat 机制（或 session start hook）每次新 session 开始时：
1. 读取 `autodream-state.json`
2. `sessions_since_last += 1`
3. 检查是否满足触发条件
4. 满足 → 在 board 的 inbox 中创建一个 AutoDream 任务（或直接触发 agent）

### 4 Phases

#### Phase 1: Orient

**读取：**
- `~/.openclaw/memory/MEMORY.md`
- `~/.openclaw/memory/*.md`（所有主题文件）
- `autodream-state.json`

**产出：** 内存中的当前记忆地图（文件列表 + 各文件摘要 + 最后修改时间）

#### Phase 2: Gather Signal

**读取：**
- `~/.openclaw/daily-notes/`（最近 7 天的笔记）
- `~/.openclaw/sessions/`（最近 session 摘要/历史）
- `.superclaw/board/done/`（最近完成的任务）

**产出：** 新信号列表（从近期活动中提取的、应该纳入记忆的信息）

#### Phase 3: Consolidate

**操作：**
- 将新信号合并到现有记忆文件
- 去重（相同信息出现在多个地方 → 合并到最佳位置）
- 修复相对日期（"昨天" → "2026-04-04"）
- 更新 MEMORY.md 索引，确保所有主题文件被引用
- 合并小主题文件（< 5 行的相关文件 → 合并）

**写入：** 更新后的 memory 文件

#### Phase 4: Prune & Index

**操作：**
- 删除空文件、过时内容（已被明确否定的结论）
- 确保 MEMORY.md 是纯索引（< 200 行），不包含实质内容
- 每个主题文件有清晰的标题和范围

**写入：** 最终的 memory 文件 + 更新 `autodream-state.json`

### Integration with Heartbeat

```
OpenClaw session start
  → 检查 autodream-state.json
  → 满足触发条件？
    ├─ no → 更新 sessions_since_last，继续
    └─ yes → 创建 AutoDream 任务 or 直接运行
             → AutoDream 4 phases
             → 更新 autodream-state.json（重置计数器）
```

AutoDream 运行方式：
- **同步模式**：在当前 OpenClaw session 中作为 skill 运行（简单，但会打断用户）
- **异步模式（推荐）**：通过 cc-delegate 启动独立 session，后台运行

### Implementation Steps

1. 设计 `autodream-state.json` schema
2. 编写 AutoDream skill：`skills/autodream/SKILL.md`
3. 实现 4 个 phase 的具体逻辑
4. 修改 session start hook 或添加新 hook 来检查触发条件
5. 编写测试：模拟触发条件、验证合并逻辑
6. 创建 `.superclaw/agents/autodream.md` 持久任务文件

### Estimated Effort

M — 约 2-3 个执行 session。核心难点在 Consolidate 阶段的合并逻辑。

## Component 5: Persistent Agent Framework

### Current State

没有持久 agent 概念。所有操作都是用户触发的、一次性的 skill 调用。

### Target State

一个通用框架，支持在后台自主运行的 agent。每个 agent 有长期存在的任务文件，由条件触发，自我报告状态。

### Framework Design

#### Agent Task File

持久 agent 的任务文件存放在 `.superclaw/agents/`，永远不归档到 `done/`。

```markdown
---
agent: autodream
type: persistent               # persistent | scheduled | event-driven
trigger:
  type: condition               # condition | cron | event
  condition: "sessions_since_last >= 5 AND hours_since_last >= 24"
state_file: .superclaw/agents/autodream-state.json
last_run: 2026-04-04T03:00:00Z
next_eligible: 2026-04-05T03:00:00Z
status: idle                    # idle | running | error
---

# AutoDream — Memory Consolidation Agent

## Purpose
定期整理 OpenClaw 记忆文件，保持记忆库整洁、无重复、索引清晰。

## Run History

| Time | Duration | Result | Notes |
|------|----------|--------|-------|
| 2026-04-04T03:00 | 45s | success | 合并 3 个文件，删除 1 个空文件 |
| 2026-04-03T02:30 | 30s | success | 更新 MEMORY.md 索引 |
```

#### Agent Registry

`.superclaw/config/agents.yaml`：

```yaml
agents:
  - name: autodream
    skill: superclaw-autodream
    type: persistent
    enabled: true
  - name: doc-sync
    skill: superclaw-doc-sync
    type: event-driven
    enabled: true
  - name: ci-watch
    skill: superclaw-ci-watch
    type: scheduled
    enabled: false  # 默认关闭，用户按需开启
```

#### Trigger Mechanism

三种触发方式：

| 类型 | 检查时机 | 示例 |
|------|---------|------|
| **condition** | 每次 OpenClaw session start | AutoDream（session 数 + 时间间隔）|
| **event** | 特定事件发生时 | Doc Sync（git commit 后检测到文档变化）|
| **scheduled** | 按 cron 表达式 | CI Watch（每 4 小时跑一次测试）|

condition 和 event 由 OpenClaw 的 session start 或 hook 触发。scheduled 需要外部 cron job 或 OpenClaw 的 heartbeat 定时检查。

#### Agent Lifecycle

```
检查触发条件
  → 满足？
    ├─ no → 记录检查时间，跳过
    └─ yes → 更新 status: running
           → 运行 agent skill
           → 更新 last_run、next_eligible、Run History
           → 更新 status: idle
           → 如果失败：status: error + 记录错误
```

### Planned Agents

#### AutoDream（详见 Component 4）

- Type: persistent / condition
- Trigger: ≥24h + ≥5 sessions
- 操作: 整理 OpenClaw 记忆文件

#### Doc Sync

- Type: event-driven
- Trigger: 检测到 `.superclaw/context/project-context.md` 与代码库不一致（git diff 有大量变化、新增/删除关键文件）
- 操作: 重新扫描项目，更新 project-context.md
- 读取: git log、目录结构、package.json 等
- 写入: project-context.md

#### CI Watch

- Type: scheduled
- Trigger: 每 N 小时（可配置，默认 4h）
- 操作: 运行项目测试套件，检测 regression
- 读取: project-context.md（获取测试命令）
- 写入: `.superclaw/reports/ci-watch-{timestamp}.md`
- 如果发现回归: 在 board inbox 创建新任务

### Implementation Steps

1. 设计 agent task file 和 registry schema
2. 编写 `agent-check.sh`：检查所有 enabled agent 的触发条件
3. 将 `agent-check.sh` 集成到 session start hook
4. 实现 AutoDream agent（Component 4）

## Component 6: Align Skill Upgrade

### Current State — SuperClaw align

当前 `skills/align/SKILL.md` 已经实现了基本的对齐流程：

- 探索项目上下文
- 一次一问澄清
- 2-3 方案对比
- 逐部分展示设计
- spec 自审
- 用户 review gate
- HARD GATE 防止跳阶

**做得好的：**
- 一次一问（已有）
- 方案对比（已有）
- 逐部分展示（已有）
- HARD GATE（已有）
- 提问顺序参考（6 个维度）

**缺失的：**
- 没有 Premise Challenge
- 没有 Forcing Questions（追问机制）
- 没有 Spec Review Loop（对抗性审查）
- 没有 Completion Status Protocol
- 提问时不够"狠"，缺少 anti-sycophancy 规则
- 没有智能跳问（smart-skip）
- 没有逃生舱（escape hatch）

### gstack office-hours 分析

gstack 的 office-hours 是一个高度结构化的对齐 skill，核心特点：

| 特性 | gstack 实现 | SuperClaw 现状 | 差距 |
|------|------------|---------------|------|
| **Mode Detection** | Startup vs Builder 两种模式 | 无分模式 | 需要适配我们的场景 |
| **Forcing Questions** | 6 个强制问题（Q1-Q6），附带 push-until-you-hear + red-flags | 6 个参考问题，无追问机制 | 需要引入追问 |
| **Anti-Sycophancy** | 5 条明确禁止语 + 5 条 pushback pattern | 无 | 直接引入 |
| **Premise Challenge** | Phase 3：挑战前提假设，输出明确的 agree/disagree 列表 | 无 | 直接引入 |
| **Smart Routing** | 按产品阶段跳问 | 无 | 适配引入 |
| **Escape Hatch** | 2 次容忍后放行 | 无 | 直接引入 |
| **Spec Review Loop** | subagent 对抗性审查，最多 3 轮 | spec 自审（自己审自己） | 引入 subagent 审查 |
| **Completion Status** | DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT | 无 | 直接引入 |
| **Cross-Model Second Opinion** | Codex/subagent 独立冷读 | 无 | 可选引入 |
| **Landscape Awareness** | 搜索外部信息 | 无 | 可选引入 |

### What to Adopt/Adapt

#### 直接引入（minimal adaptation）

1. **Premise Challenge Phase**
   - 在方案展示之前，先挑战前提假设
   - 输出 `PREMISES: 1. [statement] — agree/disagree?` 格式
   - 用户不同意 → 修正理解 → 重新循环
   - 适配：增加"这个需求是否应该由 CC 执行"的技术可行性前提

2. **Anti-Sycophancy Rules**
   - 禁止："That's an interesting approach"、"There are many ways..."、"You might want to..."
   - 要求：对每个回答都要表态，说明什么证据会改变判断
   - 适配：语言调整为中文语境

3. **Completion Status Protocol**
   - 每个 skill 结束时报告：DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT
   - 统一适用于所有 5 个 phase skill
   - 适配：status 写入任务文件的 History

4. **Escape Hatch**
   - 用户说"直接做"/"跳过问题"：容忍一次，提醒价值
   - 第二次推辞：尊重意愿，快速跳到方案阶段
   - 如果只剩 1 个问题：直接问完
   - 适配：无需修改，直接引入

5. **Spec Review Loop**
   - spec 写完后，派 subagent 做对抗性审查
   - 5 个维度：Completeness、Consistency、Clarity、Scope、Feasibility
   - 最多 3 轮迭代
   - 同一 issue 连续出现 → 作为 "Reviewer Concerns" 保留在文档中
   - 适配：reviewer subagent 需要理解我们的二层架构上下文

#### 适配引入（needs adaptation for two-layer architecture）

6. **Forcing Questions — 适配版**

   gstack 的 6 问是面向创业者的。我们需要适配为面向技术需求的版本：

   | gstack 原版 | SuperClaw 适配版 |
   |-------------|-----------------|
   | Q1: Demand Reality | **Q1: 痛点真实性** — "这个需求的触发场景是什么？多久遇到一次？不做会怎样？" |
   | Q2: Status Quo | **Q2: 现状** — "现在怎么解决的？（手动？workaround？忍着？）代价多大？" |
   | Q3: Desperate Specificity | **Q3: 最小可用** — "最小能用的版本长什么样？哪个场景先跑通？" |
   | Q4: Narrowest Wedge | **Q4: 验收标准** — "怎样算做完了？给我一个能跑的验证命令。" |
   | Q5: Observation & Surprise | **Q5: 意外情况** — "有没有试过类似方案？哪里出了意外？" |
   | Q6: Future-Fit | **Q6: 扩展性** — "这个设计 3 个月后还能用吗？会被什么东西破坏？" |

   **Smart routing by task type：**
   - feature → Q1, Q2, Q3, Q4
   - bugfix → Q2, Q4, Q5
   - refactor → Q2, Q3, Q6
   - spike → Q1, Q5

7. **Push Pattern — 追问机制**

   每个问题的第一个回答通常是"打磨过的版本"。需要追问：
   - "你说 'X'。具体到什么程度？能给个例子吗？"
   - "如果这个方案不行，Plan B 是什么？"
   - "你确定用户需要这个？还是你觉得他们需要？"

#### 不引入（unique to gstack, not applicable）

- Mode Detection（Startup vs Builder）— 我们不区分，所有都是技术需求
- Landscape Awareness（搜索外部信息）— OpenClaw 可以自行搜索，不需要内建
- Cross-Model Second Opinion — 我们有二层架构，plan 阶段 CC 本身就是"第二意见"
- Founder Signal Synthesis / YC Plea — 不适用
- Visual Sketch — 不适用于 align 阶段
- Telemetry / Contributor Mode — 不适用

### What's Unique to Our Two-Layer Architecture

gstack 是单层（人 ↔ Claude Code）。SuperClaw 是双层（人 ↔ OpenClaw ↔ Claude Code），这带来几个独特需求：

1. **技术细节过滤**：align 阶段不问用户技术细节（这是 plan 阶段 CC 的事）。gstack 会深入技术讨论，我们不会。

2. **Spec 的双重读者**：我们的 spec.md 同时给人看（确认方向）和给 CC 看（执行）。需要在人类可读性和机器可操作性之间平衡。gstack 的设计文档只给人看。

3. **Board 集成**：每个对齐步骤都要同步 board 状态。gstack 没有 board 概念。

4. **Tier 选择**：align 阶段需要确定任务的 delivery tier。这是 gstack 不需要的。

5. **Verify 命令前置**：我们要求在 align 阶段就定义 verify 命令和预期输出。gstack 不需要这个。

### Updated Align Flow

```
探索上下文 + 读 project-context.md
  → 从 board inbox 取任务 → 移到 aligning
  → Forcing Questions（一次一问，按 task type smart-route）
      → 每问追问一次（push pattern）
      → escape hatch（2 次后放行）
  → Premise Challenge（列出前提 → 用户 agree/disagree）
  → 2-3 方案 + 推荐
  → 逐部分展示设计
  → 确定 Delivery Tier
  → 定义 Verify 命令
  → 写 spec.md
  → Spec Review Loop（subagent 对抗审查，最多 3 轮）
  → 用户 review
  → Completion Status → 移到 planned → invoke plan
```

### Implementation Steps

1. 编写适配版 Forcing Questions（6 问 + smart routing + push patterns）
2. 添加 Premise Challenge phase

4. 实现 Spec Review Loop（调用 subagent）

## Component 8: Board Web UI

### Overview

一个轻量级 Web 界面，为文件驱动的看板提供可视化操作界面。文件系统始终是 source of truth，Web UI 只是一个窗口。

### Architecture

└──────────────┬────────────────────────��─┘
               │ HTTP + WS
┌──────────────┴──────────────────────────┐
│ Go Backend (single binary)              │
│ - REST API: task CRUD                   │
│ - WebSocket: real-time status push      │
│ - File watcher: fsnotify on board/      │
│ - SQLite (single user, no need for PG)  │
└──────────────┬──────────────────────────┘
               │ read/write files
┌──────────────┴──────────────────────────┐
│ .superclaw/board/ (filesystem = source  │
│ of truth)                               │
│ OpenClaw / Claude Code read/write       │
│ directly                                │
└───────────────────────────────────────���─┘
```

### Key Design Decisions

1. **File system is source of truth** — Go backend watches `.superclaw/board/` via fsnotify. When files change (agent moves a task), WebSocket pushes update to frontend. When user drags a card in UI, API call → backend mv's the file.

2. **SQLite for metadata only** — Task content lives in markdown files. SQLite caches parsed metadata for fast queries (list/filter/sort). Rebuilt from files on startup.

3. **Single binary deployment** — Go backend embeds the Next.js static build. One `superclaw-board` binary serves everything.

4. **No auth needed** — Single user on single server. Bind to localhost by default.

5. **Reference: Multica** — Borrow from `github.com/multica-ai/multica`:
   - Go backend structure: `server/internal/{handler,service,storage,realtime}`
   - WebSocket pattern: gorilla/websocket for real-time task updates
   - Frontend kanban layout: Next.js App Router with drag-and-drop columns
   - Daemon heartbeat pattern: for persistent agent status display

### What NOT to Borrow from Multica

- Multi-user auth (magic link, Google OAuth) — not needed
- S3/CloudFront file storage — all local
- pgvector — not needed
- Multi-workspace — project isolation via directories
- Resend email — not needed

### API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/board` | 获取完整看板状态（所有列 + 任务摘要） |
| `GET` | `/api/tasks` | 列出任务，支持 `?phase=&tier=&priority=` 过滤 |

| `PATCH` | `/api/tasks/:id/move` | 移动任务到新 phase，body: `{"to": "executing"}` |
| `PATCH` | `/api/tasks/:id` | 更新任务 metadata（priority、tier、assignee） |

| `GET` | `/api/agents` | 获取所有持久 agent 状态 |
| `GET` | `/ws` | WebSocket 连接，接收实时事件 |

### WebSocket Events

```typescript
// 服务端 → 客户端
type WSEvent =
  | { type: "task_moved"; taskId: string; from: string; to: string }
  | { type: "task_updated"; taskId: string; fields: Partial<Task> }
  | { type: "task_created"; task: TaskSummary }
  | { type: "agent_status"; agentName: string; status: string }
  | { type: "board_reload" }  // 大量变化，客户端应重新拉取
```

### File Watcher → WebSocket Flow

```
fsnotify 检测到 .superclaw/board/ 变化
  → 判断事件类型：
    ├─ CREATE → 解析新文件 → 更新 SQLite → 推送 task_created
    ├─ RENAME/MOVE → 判断新旧目录 → 更新 SQLite → 推送 task_moved
    ├─ WRITE → 重新解析文件 → 更新 SQLite → 推送 task_updated
    └─ DELETE → 更新 SQLite → 推送 board_reload
  → 防抖：50ms 窗口内的多个事件合并为一次推送
```
